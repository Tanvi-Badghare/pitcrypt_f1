## Overview

This document catalogues the threat actor landscape
relevant to Formula 1 telemetry infrastructure,
including known APT groups with motorsport or
automotive sector interest, relevant CVEs in
cryptographic libraries, and the adversarial kill
chain applied to the PitCrypt-F1 pipeline.

---

## Threat Actor Landscape

### TA1 — Nation-State Actors (Tier 1)

**Profile:** Sophisticated, well-resourced, patient.
Capable of supply chain attacks, zero-day exploitation,
and sustained network presence.

**Relevance to F1:** Formula 1 teams operate globally
in jurisdictions with documented state-sponsored
industrial espionage programmes. Telemetry data
contains engine performance characteristics,
aerodynamic settings, and strategic race data that
represent significant intellectual property.

**Documented incidents:**
In 2014 a major automotive manufacturer suffered
a long-term network intrusion attributed to a
nation-state actor that targeted powertrain R&D
data — directly analogous to F1 telemetry.

**Tactics:**
- Spear-phishing targeting pit wall engineers
- VPN credential theft for trackside network access
- Supply chain compromise of telemetry hardware
- Long-term persistent access before exfiltration

**Mitigations in PitCrypt-F1:**
Zero-trust IAM means network access alone is
insufficient — every packet requires cryptographic
authentication. ECDH forward secrecy limits the
value of long-term credential theft. Ed25519
non-repudiation creates evidence trail.

---

### TA2 — Competing Constructor Espionage (Tier 2)

**Profile:** Motivated, technically capable,
willing to operate near legal boundaries.
History of industrial espionage between teams.

**Relevance to F1:** The 2007 McLaren-Ferrari
espionage scandal (McLaren fined $100 million USD)
demonstrated that constructor espionage is a
documented reality in F1.

**Tactics:**
- Insider threat — hired personnel with prior
  team access
- Physical access to trackside equipment during
  race weekends
- Network sniffing on shared FIA data circuits
- Bribery of technical personnel

**Mitigations in PitCrypt-F1:**
Separate ECDH session keys per car node — Mercedes
telemetry unreadable to Red Bull encryption engine.
IAM explicitly denies cross-team telemetry access.
Physical network access without cryptographic
keys provides only encrypted ciphertext.

---

### TA3 — Opportunistic Attackers (Tier 3)

**Profile:** Unsophisticated, tools-based,
opportunistic. Target exposed services without
specific F1 knowledge.

**Relevance to F1:** Public race events attract
opportunistic attackers scanning for open ports
and unpatched services on event networks.

**Tactics:**
- Port scanning trackside network ranges
- Credential stuffing against management interfaces
- Exploitation of known CVEs in unpatched software

**Mitigations in PitCrypt-F1:**
Unknown nodes immediately rejected by IAM zero-trust.
No management interfaces exposed on telemetry ports.
All communication requires cryptographic handshake
before any data is accepted.

---

### TA4 — Insider Threat (Tier 2)

**Profile:** Current or former employee with
legitimate system access. Motivated by financial
gain, competitor recruitment, or grievance.

**Relevance to F1:** High staff turnover between
teams creates persistent insider threat risk.
Technical staff regularly move between constructors.

**Tactics:**
- Exfiltrate session keys before departure
- Modify anomaly filter thresholds to allow
  false telemetry
- Create backdoor in relay node software
- Copy audit logs before deletion

**Mitigations in PitCrypt-F1:**
Session keys are ephemeral — expire on rotation.
Key rotation means any exfiltrated key has bounded
value. Audit log is append-only — deletion requires
separate escalated access. IAM role separation means
no single insider has access to all pipeline tiers.

## Relevant CVEs

### CVE-2019-15847 — GnuPG Ed25519 Signature Bypass

**CVSS:** 7.5 (High)  
**Affected:** GnuPG < 2.2.17  
**Description:** Ed25519 signature verification
could be bypassed by providing a specially crafted
signature that passed verification despite not being
generated by the claimed key.  
**Relevance:** Directly applicable to PitCrypt-F1
Ed25519 authentication layer.  
**Mitigation:** PitCrypt-F1 uses PyCA `cryptography`
library (not GnuPG) which uses `libsodium`-backed
Ed25519 implementation — not affected by this CVE.
Library pinned to ≥41.0 in requirements.

---

### CVE-2020-1967 — OpenSSL NULL Pointer Dereference

**CVSS:** 7.5 (High)  
**Affected:** OpenSSL 1.1.1d-1.1.1f  
**Description:** TLS 1.3 signature_algorithms_cert
processing could lead to DoS via NULL pointer
dereference.  
**Relevance:** If PitCrypt-F1 were to use OpenSSL
TLS for transport, this would be relevant. Current
implementation uses custom ECDH + ChaCha20 over
raw TCP — not using TLS stack.  
**Mitigation:** Not applicable to current implementation.

---

### CVE-2022-21449 — Java ECDSA Signature Bypass (Psychic Signatures)

**CVSS:** 7.5 (High)  
**Affected:** Java SE 15-18  
**Description:** ECDSA signature verification in
Java could be bypassed by providing an (r, s)
signature where both values are zero — the
"psychic signatures" vulnerability.  
**Relevance:** Demonstrates why deterministic
signatures (Ed25519) are preferred over ECDSA.
Ed25519 uses deterministic signing — no nonce
randomness, no analogous zero-value bypass.  
**Mitigation:** PitCrypt-F1 uses Ed25519, not
ECDSA — not affected. This CVE validates ADR-001
primitive selection rationale.

---

### CVE-2016-7434 — NTP Denial of Service

**CVSS:** 7.5 (High)  
**Affected:** NTP < 4.2.8p9  
**Description:** Malformed mode 6 packet causes
NULL pointer dereference and crash.  
**Relevance:** PitCrypt-F1 timestamp validation
relies on system clock. If NTP is manipulated,
timestamp freshness checks could be bypassed.  
**Mitigation:** Timestamp checks disabled in
simulation mode (`check_timestamps: false`).
In production, NTP should be authenticated (NTPsec)
and timestamp window set conservatively.

---

### CVE-2021-3449 — OpenSSL DoS via Renegotiation

**CVSS:** 5.9 (Medium)  
**Affected:** OpenSSL 1.1.1 < 1.1.1k  
**Description:** TLS renegotiation with missing
signature algorithms extension causes NULL pointer
dereference.  
**Relevance:** Not directly applicable. Included
to illustrate the risk of using TLS stack vs custom
AEAD implementation.  
**Mitigation:** PitCrypt-F1 avoids TLS entirely —
custom ChaCha20-Poly1305 over raw TCP eliminates
TLS renegotiation attack surface.

---

### CVE-2023-38408 — OpenSSH Remote Code Execution

**CVSS:** 9.8 (Critical)  
**Affected:** OpenSSH < 9.3p2  
**Description:** Remote code execution via
ssh-agent forwarding exploitation.  
**Relevance:** Production deployment of PitCrypt-F1
nodes uses SSH for management access. Unpatched
SSH on relay or validator nodes creates RCE risk.  
**Mitigation:** OpenSSH kept patched. SSH access
restricted to specific management IPs via security
group rules. SSH agent forwarding disabled.
See `docs/CLOUD_HARDENING.md`.

## Adversarial Kill Chain

The Lockheed Martin Cyber Kill Chain applied to
a targeted attack against PitCrypt-F1:

### Phase 1 — Reconnaissance

**Objective:** Identify telemetry infrastructure
and attack surface.

**Attacker actions:**
- OSINT on F1 team IT infrastructure
- Network scanning of FIA data centre IP ranges
- Job posting analysis to identify technology stack
- Social engineering of trackside personnel

**PitCrypt-F1 exposure:**
Public GitHub repository reveals cryptographic
architecture — intentional for academic portfolio.
In production: private repository, no public
architecture documentation.

---

### Phase 2 — Weaponisation

**Objective:** Develop tools to exploit identified
vulnerabilities.

**Attacker actions:**
- Develop packet replay tool targeting sequence
  numbers
- Build MITM proxy for car → relay network segment
- Acquire 0-day for ChaCha20 implementation
  (no public 0-days exist at time of writing)

**PitCrypt-F1 exposure:**
Replay attacks require session key — not feasible
without key compromise. MITM proxy captures only
ciphertext — no plaintext without key.

---

### Phase 3 — Delivery

**Objective:** Position tooling to attack target.

**Attacker actions:**
- Physical access to trackside network during
  race weekend
- Compromise of team laptop via spear-phishing
- Rogue access point near pit lane

**PitCrypt-F1 exposure:**
Network access provides only encrypted ciphertext.
No credentials transmitted unencrypted.

---

### Phase 4 — Exploitation

**Objective:** Execute attack against pipeline.

**Attacker actions:**
- Attempt replay of captured encrypted packets
- Attempt to inject forged packets
- Attempt privilege escalation via relay node

**PitCrypt-F1 defence:**
Replay: blocked by `IntegrityChecker` and
`ValidatorSequenceChecker` independently.
Forged packets: blocked by Ed25519 verification —
no private key, no valid signature.
Relay escalation: blocked by IAM RBAC zero-trust.

---

### Phase 5 — Installation

**Objective:** Establish persistent access.

**Attacker actions:**
- Install backdoor on relay node
- Modify anomaly filter to suppress alerts
- Create rogue registered node identity

**PitCrypt-F1 exposure:**
Relay compromise does not compromise car private
keys — signature still fails at validator.
Rogue node registration requires validator
operator access — highest privilege tier.

---

### Phase 6 — Command and Control

**Objective:** Communicate with implant.

**PitCrypt-F1 defence:**
Network egress from pipeline nodes restricted to
validator endpoint only via security group rules.
Unexpected outbound connections logged by VPC
flow logs and CloudTrail.

---

### Phase 7 — Actions on Objective

**Objective:** Exfiltrate telemetry or corrupt FIA data.

**Attacker actions:**
- Exfiltrate decrypted telemetry from relay buffer
- Submit false accepted packets to FIA audit log
- Corrupt sequence numbers to invalidate race data

**PitCrypt-F1 defence:**
Relay policy denies plaintext storage. Ephemeral
plaintext only during processing window.
Audit log is append-only — false records require
validator-level access.
Session key rotation bounds any captured data.
